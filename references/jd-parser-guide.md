# 职位描述解析指南 (JD Parser Guide)

## 概述

本文档指导面试官技能如何从招聘广告、职位描述（JD）中提取结构化信息，
并利用这些信息个性化定制面试内容。

---

## 1. 输入源处理

### 1.1 招聘链接

用户可能粘贴以下平台的链接：

| 平台 | URL 特征 | 处理方式 |
|-----|---------|---------|
| LinkedIn | `linkedin.com/jobs/view/` | WebFetch 获取页面 |
| Indeed | `indeed.com/viewjob` | WebFetch 获取页面 |
| Glassdoor | `glassdoor.com/job-listing/` | WebFetch 获取页面 |
| Boss直聘 | `zhipin.com/job_detail/` | WebFetch 获取页面 |
| 拉勾 | `lagou.com/jobs/` | WebFetch 获取页面 |
| 牛客 | `nowcoder.com/` | WebFetch 获取页面 |
| 猎聘 | `liepin.com/job/` | WebFetch 获取页面 |
| 脉脉 | `maimai.cn/` | WebFetch 获取页面 |
| GitHub Jobs | `github.com/` | WebFetch 获取页面 |
| 公司官网 | 各种 | WebFetch 获取页面 |

**WebFetch 提取指令模板：**
```
提取该招聘页面中的以下信息：
1. 职位名称
2. 公司名称
3. 工作地点
4. 薪资范围（如有）
5. 职位级别/年限要求
6. 必备技能要求（Required/必须）
7. 加分技能（Preferred/优先/Nice-to-have）
8. 核心职责列表
9. 教育要求
10. 技术栈要求（编程语言、框架、工具）
11. 公司/团队描述
12. 其他要求或福利信息
请尽量完整提取，保留原文关键词。
```

**如果 WebFetch 失败（某些平台可能限制抓取）：**
- 告知用户该链接无法直接获取
- 请用户手动复制粘贴 JD 文本
- 或请用户截图 JD 页面

### 1.2 粘贴文本

直接解析文本内容，识别 JD 结构。JD 文本通常包含以下标准段落：
- About the Role / 职位描述
- Responsibilities / 工作职责
- Requirements / 任职要求
- Nice to Have / 加分项
- Benefits / 福利待遇

### 1.3 截图

使用 Read 工具查看图片，OCR 提取文字内容后按文本模式处理。

## 2. JD 六维透镜解读

### 2.1 频率分析

统计 JD 中关键词出现频率：

```
频率 ≥ 3次 → 核心考察重点，面试中必须覆盖
频率 = 2次 → 重要关注点
频率 = 1次 → 一般要求
```

**示例：** 如果一个 JD 中 "distributed systems" 出现了4次，"Kubernetes" 出现3次，
说明分布式系统和 K8s 是该岗位的核心能力要求，面试中应重点考察。

### 2.2 顺序与强调分析

- JD 中排在最前面的要求 = 最重要的要求
- 出现在"Required"而非"Nice-to-have"中 = 硬性门槛
- 使用"strong"、"deep"、"extensive"等修饰词 = 高标准要求
- 使用"familiar"、"exposure"等修饰词 = 了解即可

### 2.3 动词语义分析

| JD 中的动词 | 暗示 | 面试考察方向 |
|-----------|------|------------|
| Build / 构建 | 从零到一 | 系统设计能力、架构决策 |
| Optimize / 优化 | 改进现有系统 | 性能分析、问题诊断能力 |
| Scale / 扩展 | 处理增长 | 分布式系统、可扩展性设计 |
| Lead / 领导 | 团队管理 | 技术领导力、沟通协调 |
| Mentor / 指导 | 培养他人 | 知识传递、代码review能力 |
| Design / 设计 | 架构层面 | 系统设计、技术方案评估 |
| Maintain / 维护 | 运维导向 | 问题排查、运维经验 |
| Migrate / 迁移 | 技术升级 | 渐进式改造、风险控制 |
| Collaborate / 协作 | 跨团队 | 沟通能力、跨职能合作 |

### 2.4 弦外之音解读

| JD 表述 | 可能的真实含义 | 面试启示 |
|--------|-------------|---------|
| "Fast-paced environment" | 可能人手不足，需要快速迭代 | 考察抗压能力和优先级判断 |
| "Wear many hats" | 需要全栈能力，角色边界模糊 | 考察技术广度和适应力 |
| "Self-starter" / "自驱" | 较少手把手指导 | 考察主动性和独立解决问题能力 |
| "Ambiguity" / "不确定性" | 需求不清晰是常态 | 考察在不确定条件下的决策能力 |
| "Cross-functional" | 需要大量跨团队沟通 | 考察沟通和影响力 |
| "Greenfield project" | 全新项目 | 考察从0到1的经验 |
| "Technical debt" | 遗留系统问题多 | 考察重构经验和耐心 |
| "Data-driven" | 重视数据决策 | 考察数据分析和实验能力 |

### 2.5 缺失信号分析

注意 JD 中刻意没提到的内容：

- 没提到 work-life balance → 可能加班文化
- 没提到具体技术栈 → 可能技术选型灵活，考察广度
- 没提到团队规模 → 可能是新团队
- 只有高级要求没有基础要求 → 寻找资深人才
- 没提到晋升路径 → 可能是扁平结构

### 2.6 公司/团队背景推断

从 JD 中的公司描述和团队信息推断：
- 公司阶段（初创/成长/成熟）
- 团队规模和成熟度
- 技术文化倾向
- 可能的面试风格（如大厂偏算法、创业公司偏实战）

## 3. 简历 × JD 交叉匹配算法

### 3.1 匹配分类

将简历中的每项经验与 JD 要求进行映射：

```
匹配度分级：
├── ✅ 强匹配 (Strong Fit)
│   └── 简历经验直接对应 JD 的 Required 项
│       例：JD要求"3年Python经验"，简历有"4年Python开发"
├── ⚡ 可迁移 (Transferable)
│   └── 不直接匹配但存在合理关联
│       例：JD要求"Kubernetes"，简历有"Docker + 容器编排经验"
├── ⚠️ 差距 (Gap)
│   └── JD 要求但简历中完全缺乏的能力
│       例：JD要求"分布式系统设计"，简历无相关经验
├── 🎯 超出 (Exceeds)
│   └── 简历中超出 JD 要求的能力
│       例：JD要求"熟悉ML"，简历有"发表过ML论文"
└── ❓ 待验证 (To Verify)
    └── 简历声称具备但需面试验证
        例：简历写"性能提升200%"但缺少具体细节
```

### 3.2 面试策略映射

| 匹配类别 | 面试策略 | 问题设计 |
|---------|---------|---------|
| 强匹配 | 深度验证 | 三层追问，验证是真实深度还是表面了解 |
| 可迁移 | 关联验证 | "你在 X 的经验如何应用到 Y 场景？" |
| 差距 | 潜力探测 | 提供学习场景题，看候选人的推理和学习能力 |
| 超出 | 展示机会 | 给空间展示优势，同时验证可信度 |
| 待验证 | 重点追问 | 事实层追问：具体数据、方法、对比基准 |

## 4. JD 导向的评估报告附加章节

当有 JD 信息时，评估报告额外增加：

```markdown
## 岗位匹配度分析

### 目标岗位：{公司} - {职位}

### 匹配度总评：{高/中/低}

### 必备要求匹配
| JD 要求 | 匹配状态 | 说明 |
|--------|---------|------|
| {要求1} | ✅ 匹配 | 有4年相关经验 |
| {要求2} | ⚡ 可迁移 | 有类似但不完全匹配的经验 |
| {要求3} | ⚠️ 差距 | 缺乏相关经验 |

### 加分项匹配
| JD 加分项 | 匹配状态 | 说明 |
|----------|---------|------|
| {加分1} | ✅ 匹配 | ... |

### 针对该岗位的面试准备建议
1. {最重要的准备方向}
2. {第二重要的准备方向}
3. {差距项如何短期弥补}
```
